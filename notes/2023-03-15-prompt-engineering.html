提示工程：把语言模型能力变成稳定工作流

主题：提示设计、推理、检索、工具调用

对应原文：Lilian Weng, Prompt Engineering（2023）

这篇文章在回答什么

同一个模型为什么会因为提问方式不同而给出截然不同的结果？这篇综述把提示视为一种输入侧的行为接口：它不改变模型权重，却能改变模型接收到的任务、约束、上下文和可用动作。

原作者的论证骨架

  1. 基础提示：零样本、少样本与指令提示是最直接的三种控制手段。少样本的关键不只在“给例子”，还在示例是否相关、排序是否合适。
  2. 显式推理：Chain-of-Thought 与 self-consistency 通过暴露中间步骤、或对多条推理路径做聚合，提高需要多步计算的问题表现。
  3. 自动化设计：提示可以被搜索、改写或由另一个模型优化，不必完全依赖人工试错。
  4. 能力外接：检索、程序执行和外部 API 把“只根据参数记忆回答”的模型，变成能读取事实、运行计算和调用工具的系统。

整理者归纳

提示工程不是“写一句更聪明的话”，而是在设计四个边界：

  • 任务边界：明确输入、输出格式、完成标准与不做什么。
  • 证据边界：要求回答依据已给材料；需要实时事实时接入检索并展示出处。
  • 推理边界：高风险或复杂任务允许分解、复核和工具计算，简单任务不强制冗长思考。
  • 动作边界：把能否调用工具、何时请求确认、失败如何收敛写成工作流，而不是留给模型临场猜测。

因此，提示本身只是控制面的一部分。真正可复用的能力来自“提示 + 上下文 + 工具 + 评估”的闭环。

可直接采用的做法

  1. 先写验收标准，再写提示：结果应该包含哪些字段、哪些引用、怎样才算失败。
  2. 少样本示例只保留最能覆盖边界条件的 2 到 5 个；示例重复会挤占有效上下文。
  3. 需要事实性时，要求先检索再归纳，并把“证据”和“结论”分开输出。
  4. 把稳定的提示固化成模板或任务说明，而不是散落在聊天记录里。

与知识库的关系

  • LLM Agent：提示负责定义 agent 的局部行为，工作流负责把局部行为串起来。
  • 幻觉治理：检索、引用和拒答条件是事实性提示设计的核心。
  • Harness Engineering：当提示、文件和工具被一起编排时，提示工程升级为运行环境设计。
