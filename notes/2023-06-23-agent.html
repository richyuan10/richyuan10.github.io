LLM Agent：规划、记忆、工具三件事如何协作

主题：Agent 架构、任务分解、检索记忆、工具调用

对应原文：Lilian Weng, LLM Powered Autonomous Agents（2023）

这篇文章在回答什么

语言模型如何从“给一句输入、回一句输出”变成能完成多步任务的 agent？文章给出的经典答案是：在模型外建立规划、记忆和工具调用三个组件，并让模型在循环中决定下一步。

原作者的论证骨架

  1. 规划：通过任务分解把长任务拆成可执行步骤；通过自我反思对中间答案进行批评和修订。
  2. 记忆：短期记忆保存当前上下文，长期记忆用向量检索等方式把相关历史重新带回当前任务。
  3. 工具：搜索、代码执行、API 与软件操作扩展了模型的感知和行动能力。
  4. 案例与挑战：文章用科学发现 agent、生成式社会模拟等案例说明架构，同时指出有限上下文、可靠性、成本和安全仍是核心难题。

整理者归纳

“Agent”不是一个更大的 prompt，而是一个带状态的控制循环。一个可落地的 agent 至少要回答四个问题：

  • 当前任务的状态保存在哪里？
  • 下一步由谁决定，模型还是确定性规则？
  • 工具调用的输入、权限和失败处理是什么？
  • 什么时候停止，怎样知道结果可靠？

规划、记忆和工具不是必选套餐。任务很短时，只需工具调用；需要跨轮积累时再引入记忆；只有确实存在分支决策或长链条依赖时，才值得增加规划与反思循环。

实践检查单

  1. 将任务状态写入文件或数据库，不能只依赖对话窗口。
  2. 每个工具定义输入 schema、超时、重试条件和可验证的输出。
  3. 给循环设预算：最大步数、最大工具调用数、最大上下文长度。
  4. 把“完成”定义为可检验的产物，例如测试通过、引用齐全或结构化字段完整。

与知识库的关系

  • 提示工程：提示约束单步决策，agent 架构约束多步执行。
  • Harness Engineering：文件系统、子任务和后台作业是 agent 运行环境的具体化。
  • 幻觉治理：工具结果与记忆检索都需要来源、时效和置信度管理。
