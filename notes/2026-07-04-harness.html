Harness Engineering：把模型放进可积累、可验证的工作环境

主题：Agent 运行环境、持久记忆、工作流、持续改进

对应原文：Lilian Weng, Harness Engineering for Self-Improvement（2026）

这篇文章在回答什么

当基础模型能力已足够时，为什么实际产出仍高度依赖它所处的运行环境。文章把 harness 定义为围绕模型的工作流、文件、工具、子任务和评估机制，并讨论它怎样支持持续改进。

原作者的论证骨架

  1. 设计模式：工作流自动化、把文件系统作为持久记忆、子 agent 与后台作业，构成可重复的外部执行环境。
  2. 案例：编码 agent 的 harness 展示了模型核心能力与任务环境如何分层协作。
  3. 优化对象：上下文工程、工作流设计、自我改进、进化式搜索与模型权重联合优化，都是可被系统化调整的部分。
  4. 挑战：改进需要可靠评估、成本控制和安全边界，不能只依赖“循环运行更久”。

整理者归纳

Harness 是让模型能力变成组织能力的地方。它把原本一次性的对话变成可回放、可检查、可重启的系统：

  • 文件承接长期记忆和中间产物；
  • 工作流定义任务顺序、失败处理与停止条件；
  • 评估决定什么样的结果值得保留；
  • 人的确认点控制不可逆动作和知识质量。

对个人知识库来说，这正是关键：原始快照、来源登记、wiki 页面、操作日志和待确认列表都应该是 harness 的一部分，而不是临时聊天的副产品。

核心命题：提升系统，不只提升模型

文章的真正转向是把“递归自我改进”从一个容易让人联想到模型自己改权重的概念，落到更近的工程路径：模型先改善训练管线与部署系统，再让下一轮模型或同一个模型在任务上表现更好。

这里的关键对象是 harness。它位于基础模型和真实任务之间，决定模型：

  • 看见什么上下文；
  • 能调用什么工具，以及权限边界；
  • 怎样把长任务拆成步骤；
  • 哪些中间产物被保存；
  • 什么结果算成功，什么失败会被保留和复盘。

这意味着，同一底座模型在不同 harness 中可以呈现出截然不同的稳定性。模型能力决定上限的一部分；harness 决定能力是否能在长任务、真实工具和反复迭代中被可靠释放。

Agent 与 Harness 的区别

层次 要解决的问题 典型组件
Agent 当前这一步怎样决定和行动 规划、记忆、工具调用、反思
Harness 整个任务怎样持续、恢复、评估和演化 工作流、文件状态、任务队列、权限、测试、日志、人工确认

一个 agent 可以是一次性的：读 prompt，调用一次工具，输出答案。Harness 则更像运行时或操作系统：它不替模型“想”，但为模型提供可观察的状态、受控的动作空间和可回归的任务循环。

这也是为什么文章把 coding agent 当作主要案例。代码仓库本身提供了文件、版本历史、测试、构建日志和 diff；这些外部工件让 agent 不必把一切塞进上下文窗口，也让人能判断它到底做了什么。

三个基础设计模式

1. 工作流自动化：把“会回答”改成“会验证地推进”

原作者给出的循环可以概括为：规划 -> 执行 -> 观察或测试 -> 改进 -> 再执行。它和静态 prompt 的差别在于，工具输出与失败记录会改变下一步，而不是只让模型一次性生成更长的答案。

对工程任务，这意味着“写代码”之后必须有测试、日志或可检查的工件；对研究任务，则必须有实验结果、证据追踪或失败原因。没有观察和验证的循环只是重复生成，不是改进。

2. 文件系统作为持久记忆：把上下文从聊天记录迁移到工件

长任务的日志、代码差异、实验结果和论文笔记会很快超过上下文窗口。文章强调，系统不应把全部历史不断拼接进 prompt，而应把持久状态外置为文件，由模型在需要时检索、更新和总结。

这背后有两个好处：

  1. 任务中断后可以恢复，不依赖某个会话还在；
  2. 失败路径也能被保存，后续系统可以分析“为什么失败”，而不是只看到最终答案。

但文件不是天然的记忆。要有目录约定、命名、摘要、索引和版本边界；否则只是把上下文污染转移到磁盘。

3. 子 agent 与后台任务：并行，但必须可见

并行适合搜索多种假设、运行独立实验或把局部任务隔离开。文章提出父 agent 需要承担一个小型进程管理器的职责：启动任务、检查日志、取消失败作业，并把结果合并回主线。

这里最容易被忽略的约束是可审计性。如果子任务的结论只存在于瞬时聊天上下文，主 agent 无法可靠复盘，用户也无法知道结论来自什么过程。正确的并行化应产出状态记录、日志和可合并的工件，而不只是更多文本。

优化阶梯：从 prompt 到“优化优化器”

文章给出了一条很有价值的抽象阶梯：

指令 / prompt -> 结构化上下文 -> 工作流 -> harness 代码 -> 优化 harness 的代码

每向上一层，搜索空间更大，潜在收益也更大，但评估难度和安全风险同步上升。

  • 优化 prompt，改动小、试验快，但能力边界有限；
  • 优化上下文，开始决定哪些信息长期保留、如何检索；
  • 优化工作流，开始决定不同 agent、工具和验证步骤怎样编排；
  • 优化 harness 代码，系统可以改写自己的执行机制；
  • 优化优化器，系统连“如何提出下一轮改进”都成为搜索对象。

文章中 ACE、MCE、Meta-Harness、ADAS、AFlow 等工作都可放在这条阶梯上理解。它们不是同一种方法，而是在不同层级上把原本人工编写的机制变成可评估、可搜索的对象。

自我改进的真实闭环

“自己改自己”容易被误解成无约束的自动进化。文章讨论的更可信闭环是：

  1. 采集轨迹：运行当前 harness，保存任务结果、工具调用和失败痕迹；
  2. 挖掘弱点：按可验证的失败模式聚类，而不是只看“超时”或“没产出”这类表面标签；
  3. 提出小改动：限定可编辑区域，优先处理反复出现、因果较清楚的问题；
  4. 回归验证：同时检查已知失败是否修复，以及未见任务是否退化；
  5. 选择性合并：只有通过验证的改动进入主 harness，失败候选也要留档。

这比“让 agent 阅读自己的日志，然后自由改 prompt”严格得多。它把改进变成一个软件工程的变更管理问题：可定位、可测试、可回滚。

为什么评估和权限必须在循环外

这是文章最重要、也最容易被宣传叙事掩盖的一点。若 harness 可以修改评估器、扩大自己的权限或只在训练集上自评，它会迅速进入 Goodhart 式优化：通过满足评分信号而非真实目标来“变好”。

原文特别提醒：自我改进系统的可编辑面必须被设计；权限与安全层要置于进化循环之外。实际含义是：

  • 测试集与关键评估规则不能由被评系统自行改写；
  • 有外部副作用的工具必须有明确权限与人工确认点；
  • 新版本至少要同时通过针对已知问题的测试和独立的保留测试；
  • 失败、负结果和被拒绝的修改不能被悄悄丢弃。

这也解释了为什么文章对“自动科研”保持审慎：能生成论文不等于发现成立；能跑完实验不等于实验回答了正确的问题。

放到拾贝：这篇文章怎样改变我们的知识库

这篇文章不是一个遥远的 agent 研究话题，它直接规定了拾贝的工作方式：

Harness 原则 拾贝中的实现
持久状态 raw/ 保存原始快照，sources/ 保存来源登记，wiki/ 保存加工结果
可恢复工作流 ingest、编译、构建与发布由脚本执行，不依赖某次聊天仍然存在
受控修改 论文在用户确认前只进候选清单，不写入正式知识结论
外部评估 原文链接和本地快照用于核验；站内页面明确区分原作者与整理者
避免奖励黑客 不以“生成了多少页面”为指标，而看引用、可读性和错误率

因此，本站从“把外部链接排成目录”改为“只展示已加工笔记”，并不是单纯的 UI 调整，而是 harness 设计的一次修正：未经正文处理和证据核验的材料不进入阅读层。

读这篇文章时应保留的判断

  1. Harness 很重要，但不是替代模型智能。 原文引用的自我改进实验也显示，较弱底座模型未必能通过递归结构获得持续增益。
  2. 自动可评估任务最适合这条路线。 代码测试、内核优化、算法题等有明确分数；研究品味、长期维护性、用户价值则难得多。
  3. 长程记忆不是无限上下文。 文件化状态解决的是可恢复和可检索，仍需摘要、选择、去重与版本管理。
  4. 更强的自修改能力要求更强的治理。 可编辑面越大，评估隔离、权限控制、回归测试和人工监督越不能省。

可直接采用的做法

  1. 为每个来源保留不可变原始快照，并将加工结果与快照分层保存。
  2. 把“抓取、整理、校验、发布”写成可重复脚本，任何一步失败都能从状态恢复。
  3. 在高影响写入前保留人工确认，例如论文是否纳入、结论是否升级为正式页面。
  4. 以用户阅读质量、引用完整度和错误率评估流程，不以生成篇数作为主要指标。

与知识库的关系

  • LLM Agent：agent 是执行者，harness 是它执行任务的环境与约束。
  • 提示工程：提示模板应与文件、工具和验收规则一起版本化。
  • 奖励黑客：评估与人工确认是防止系统只优化表面产出的必要护栏。
